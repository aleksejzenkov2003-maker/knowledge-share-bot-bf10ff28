

## Цель

Привести опубликованный шаблон **«Отказы по ТЗ»** (`87cb82c1…`) в полное соответствие с регламентом из загруженного DOCX: 12 функциональных шагов + двойной финальный результат (внутреннее юридическое заключение + проект возражения в Роспатент). Шаблон станет «боевым» — каждый шаг будет иметь чёткий вход, действия, выход, критерии качества и встроенную QC-проверку перед передачей дальше.

## Что есть сейчас и что не так

В текущем шаблоне 9 узлов, но регламент требует 12 + два финальных артефакта. Отсутствуют важные шаги:
- Шаг 1 «Приём и формальная проверка пакета» (читаемость, OCR, реквизиты, крайний срок) — реализовано фрагментарно;
- Шаг 3 «Сведения о заявителе и доказательства использования» — нет вообще;
- Шаг 5 «Анализ заявленного обозначения и перечня товаров/услуг» — нет;
- Шаг 11 «Сводные правовые блоки по каждому основанию» в строгой 9-польной структуре — нет;
- Шаг 12 «Стратегия A/B/C + сводная матрица + дорожная карта» — нет;
- Финальный шаг сборки в **два** документа (заключение + проект возражения) — отсутствует;
- Нет нод `quality_check` между ключевыми шагами и нет `result` нод с шаблоном итогового документа.

Также не хватает связей `condition`-аварийной эскалации (при некачественном OCR / неоднозначной квалификации).

## Целевая структура шаблона (этапы и узлы)

```text
Этап 0 — Вход
  [input] Ввод данных (обновлённая форма)

Этап 1 — Приём
  [agent] 1. Парсер пакета и OCR-проверка
  [quality_check] QC-полнота входа

Этап 2 — Формальная проверка ОА
  [agent] 2. Формальная проверка уведомления (реквизиты, срок, нормы)
  [quality_check] QC-формалки

Этап 3 — Контекст заявителя и обозначения
  [agent] 3. Заявитель и доказательства использования
  [agent] 5. Анализ обозначения и перечня товаров/услуг

Этап 4 — Юридическая квалификация и противопоставления
  [agent] 4. Классификатор оснований отказа (по пунктам ст. 1483)
  [agent] 6. Юр. квалификация позиции экспертизы
  [agent] 7. Выгрузка/верификация противопоставлений
  [agent] 8. Сравнение обозначений и однородности
  [agent] 9. Статус и активность правообладателей
  [quality_check] QC-аналитики

Этап 5 — Практика и аргументация
  [agent] 10. Практика СИП/ППС/Роспатента
  [agent] 11. Сводные правовые блоки (9-польная структура по каждому основанию)
  [quality_check] QC-аргументации

Этап 6 — Стратегия
  [agent] 12. Стратегия A/B/C, матрица, дорожная карта

Этап 7 — Итог
  [agent] Сборка внутреннего юридического заключения
  [agent] Сборка проекта возражения в Роспатент
  [result]  Итог: два документа (заключение + возражение)
```

Все рёбра — `passthrough` по умолчанию, между QC и следующим этапом — условие `qc_passed == true`, иначе ребро эскалации возвращает на доработку соответствующего шага.

## Конкретные изменения

### 1. Обновить форму ввода (узел `input`)
Поля по регламенту шага 1:
- `designation` (текст) — заявленное обозначение;
- `application_number` (номер заявки);
- `applicant_name` (заявитель);
- `oa_text` (textarea) — текст уведомления Роспатента;
- `oa_date` (текст/дата) — дата уведомления;
- `mktu_classes` (текст) — классы МКТУ;
- `goods_services_raw` (textarea);
- `opposed_marks_raw` (textarea) — список противопоставлений из ОА;
- `client_priorities` (textarea) — что критично для бизнеса (для шага 5/12);
- `attachments` (file, multiple) — ОА в PDF/DOCX, изображение знака, переписка.

### 2. Реиспользовать существующих агентов и добавить недостающих
Уже есть и подходят:
- `(ТЗ-отказ 1) Парсер ОА`, `(ТЗ-отказ 2) Классификатор`, `(ТЗ-отказ 3) Эксперт охраноспособности`, `(ТЗ-отказ 4) Эксперт сходства`, `(ТЗ-отказ 5) Статус противопоставлений`, `(ТЗ-отказ 6) Поиск практики`, `(ТЗ-отказ 7) Стратег`, `(ТЗ-отказ 8) Составитель ответа`, `(ТЗ-отказ 9) QC-ревьюер`.

Добавить (через миграцию `INSERT INTO chat_roles`):
- **(ТЗ-отказ 0) Координатор-приёмщик** — модель `claude-sonnet-4-6`, делает чек-лист полноты входа, проверку читаемости, расчёт крайнего срока с пометкой «требует верификации юристом».
- **(ТЗ-отказ 3a) Аналитик заявителя** — модель `sonar-pro`, web-search включён; собирает доказательства использования, шкала «официальный документ / независимый источник / собственный ресурс».
- **(ТЗ-отказ 5a) Аналитик обозначения и перечня** — модель `claude-sonnet-4-6`; доминирующий/слабый/описательный элемент, ядро перечня, резервные варианты сужения.
- **(ТЗ-отказ 11) Сборщик правовых блоков** — модель `claude-opus-4-6` (или `claude-sonnet-4-6`); строго 9-польная структура по каждому основанию.
- **(ТЗ-отказ 12) Редактор внутреннего заключения** — модель `claude-sonnet-4-6`; полная структура из раздела 5 регламента.

- **(ТЗ-отказ QC-аналитика)**, **(ТЗ-отказ QC-аргументации)** — `claude-sonnet-4-6`; узкоспециализированные QC-промпты по критериям регламента.

Каждому новому агенту — `system_prompts` запись с дословными «Действиями» и «Критериями качества» из соответствующего шага регламента + жёсткие запреты («запрещено формулировать “знак не используется” без надлежащих доказательств» и т. п.).

### 3. Перестроить узлы шаблона
Через миграцию:
- удалить старые `workflow_template_steps` и `workflow_template_edges` шаблона `87cb82c1…`;
- вставить 16 новых шагов согласно структуре выше с правильными `stage_group`, `stage_order`, `position_x/y` (раскладка по горизонтальным «дорожкам» по этапам);
- проставить `quality_check_agent_id` для трёх QC-нод;
- настроить `script_config` QC-нод правилами `not_empty` для ключевых полей выхода предыдущего шага (по образцу мем-правила Workflow Quality Check Rules);
- вставить рёбра: основной поток `passthrough` + рёбра-эскалации с `conditions: [{field: "qc_passed", operator: "eq", value: false}]`, ведущие назад на блок-источник.

### 4. Финальный узел `result`
Добавить `node_type='result'` с `result_assembly_mode='combined'`. Промпт-сборка использует разделители (по образцу мем-правила Dual Workflow Output):
```
===ЗАКЛЮЧЕНИЕ_ВНУТРЕННЕЕ===
{внутреннее юридическое заключение}
===ПРОЕКТ_ВОЗРАЖЕНИЯ===
{проект возражения для Роспатента}
```
Это позволит фронту разнести их в две вкладки результата и при необходимости экспортировать через `generate-kp-docx` (та же edge-функция уже умеет принимать структурированный markdown).

### 5. Системные промпты — что строго закрепить
- ссылаться только на конкретные пункты ст. 1477–1483, 1492, 1512 ГК РФ и адресные разделы Руководства Роспатента;
- по каждому основанию отказа — отдельный блок, без смешения;
- по каждому противопоставлению — отдельный блок;
- при нехватке данных — заполнять поле `missing_data`, а не делать вывод;
- срок ответа — всегда с пометкой «требует верификации юристом»;
- использование/неиспользование — только формулой «обнаружены признаки отсутствия выявленного использования по открытым источникам».

### 6. Карточка данных шага
Зафиксировать в `output_schema` каждого `agent`-узла единый JSON-каркас из раздела 3 регламента:
`{ step_id, agent, input, facts, law, practice, risks, missing_data, output, qc }`. Это даёт стабильный контракт для сборщика 11→12→финал.

### 7. Условные ветки и эскалация (раздел 4.2 регламента)
Эскалация на человека реализуется как условное ребро от QC-нод к финальному узлу с пометкой `escalation=true`, если:
- OCR качество низкое (`ocr_quality<0.7`),
- основания не классифицируются однозначно (`classification_confidence<0.6`),
- противоречивая практика (`practice_conflict=true`),
- слабая доказательная база (`evidence_strength=low`).
Эти поля QC-агенты обязаны заполнять в выходном JSON.

## Технические детали реализации

- Новые `chat_roles` и `system_prompts` — одна миграция.
- Перестройка `workflow_template_steps` / `workflow_template_edges` для `87cb82c1…` — отдельная миграция (`DELETE … WHERE template_id=…` + массовый `INSERT`).
- Версия шаблона (`workflow_templates.version`) увеличивается на +1, статус остаётся `published`.
- Никаких изменений в edge-функциях не требуется — `workflow-step-execute` уже поддерживает `quality_check`, `condition`-рёбра, `result_assembly_mode='combined'` и stage-grouping.
- Никаких клиентских изменений не требуется — редактор воркфлоу и Stepper уже умеют рендерить новые этапы по `stage_group`/`stage_order` (см. мем-правило Workflow Stage Grouping).
