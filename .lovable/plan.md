

# Исправление RAG-поиска по МКТУ: семантический разрыв

## Проблема

Пользователь спрашивает: *"Подбери классы МКТУ для вендингового оборудования"*. Слово **"вендинг"** отсутствует в МКТУ — вместо него используются формулировки вроде "автоматы торговые", "аппараты для приготовления напитков", "машины для продажи товаров".

**Что происходит сейчас:**
1. Synonym expansion для "мкту" добавляет слова: `класс, классификация, товары, услуги`
2. FTS находит 100 **случайных** чанков (слова "класс" и "товары" есть в каждом из 925 чанков)
3. Reranker получает случайную выборку и не может найти нужные классы, потому что правильных чанков в выборке просто нет

**Корень проблемы:** FTS-поиск не может найти семантически близкие понятия ("вендинг" → "торговые автоматы", "аппараты для розлива напитков").

## Решение: LLM Query Expansion для МКТУ

Добавить шаг **LLM-based query expansion** перед FTS-поиском. Когда запрос связан с подбором классов МКТУ, использовать быструю модель (Gemini Flash) чтобы сгенерировать список конкретных слов/фраз из словаря МКТУ, которые могут быть релевантны.

### Архитектура

```text
User Query: "вендинговое оборудование"
         │
         ▼
┌─────────────────────────┐
│ LLM Query Expansion     │  ← Gemini Flash (быстро, дёшево)
│ "Разложи на термины     │
│  МКТУ: автомат торговый, │
│  аппарат для напитков,   │
│  программное обеспечение,│
│  ремонт машин..."        │
└─────────────────────────┘
         │
         ▼
┌─────────────────────────┐
│ FTS Search с expanded   │  ← Теперь находит правильные чанки
│ keywords + OR query     │
└─────────────────────────┘
         │
         ▼
┌─────────────────────────┐
│ Claude Reranker         │  ← Работает с правильными кандидатами
└─────────────────────────┘
```

### Изменения

**Файл: `supabase/functions/chat-stream/index.ts`**

1. **Определять МКТУ-контекст**: если роль содержит "мкту" в имени или системный промпт содержит "МКТУ", активировать расширенный поиск

2. **LLM Query Expansion**: вызвать Gemini Flash с промптом:
   ```
   Пользователь описал деятельность: "{query}".
   Сгенерируй 20-30 конкретных терминов из классификатора МКТУ (товары и услуги),
   которые могут относиться к этой деятельности. Только термины, по одному на строку.
   ```
   Это займёт ~1-2 секунды и даст конкретные термины для FTS.

3. **Расширенный keyword search**: использовать сгенерированные термины как keywords для `keyword_search` RPC (ILIKE-based) — он найдёт чанки с "автомат торговый", "аппарат для приготовления", и т.д.

4. **Объединение результатов**: объединить FTS-результаты и keyword-результаты от LLM expansion, дедупликация, затем отправить в reranker

### Почему именно так

- **Не нужна векторная БД**: embeddings для МКТУ дали бы похожий результат, но требуют масштабной переиндексации
- **Gemini Flash — дёшево и быстро**: 1-2 сек, минимальная стоимость
- **Обратно совместимо**: для не-МКТУ запросов всё работает как раньше
- **Решает корневую проблему**: мост между бытовым языком пользователя и формальным словарём МКТУ
