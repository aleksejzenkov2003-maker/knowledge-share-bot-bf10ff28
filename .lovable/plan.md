

# Глубокий анализ проблем системы цитирования

## Архитектура текущей системы

```text
┌─────────────────────────────────────────────────────────────────────────────────┐
│  1. Поиск документов (RAG)                                                       │
│     FTS/Keyword Search → Re-ranking (Claude) → rankedChunks (до 10 чанков)      │
└─────────────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  2. Формирование контекста для Claude                                            │
│     ragContext[] = "[1] DocName | Section\nContent..."                          │
│                    "[2] DocName | Section\nContent..."                          │
└─────────────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  3. Инструкция Claude                                                            │
│     "ОБЯЗАТЕЛЬНО указывай источники в формате [номер] для каждого утверждения"  │
│     ❌ НЕТ правил: как различать несколько утверждений из одного источника     │
│     ❌ НЕТ правил: что делать если утверждение из нескольких источников        │
└─────────────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  4. Генерация ответа Claude                                                      │
│     "Согласно правилам [1], товарный знак должен... [1]"                        │
│     ❌ Claude ставит [1] произвольно, без привязки к конкретному фрагменту     │
└─────────────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  5. UI отображение                                                               │
│     - "Источники" = ragContext[] (текстовые чанки с [N] в начале)              │
│     - "Цитаты" = citations[] (отфильтрованные по [N] в ответе)                  │
│     ❌ Оба показывают ТЕ ЖЕ данные, просто в разных вкладках                   │
│     ❌ НЕТ связи "это утверждение → эта цитата"                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

## Корневые причины каждой проблемы

### Проблема 1: Дублирование номера сноски [1] для разных утверждений

**Местоположение**: `chat-stream/index.ts`, строки 729-744

**Причина**: Инструкция Claude слишком простая:
```typescript
instructions = `
2. ОБЯЗАТЕЛЬНО указывай источники в формате [номер] для каждого утверждения
`;
```

Claude интерпретирует это как "ставь [1] везде, где использовал источник 1", а не "ставь уникальную сноску для каждого факта".

**Корень**: Нет концепции "inline citation ID" — уникального идентификатора для каждого конкретного утверждения в тексте.

---

### Проблема 2: Несоответствие нумерации ссылок количеству источников

**Местоположение**: 
- `chat-stream/index.ts`, строки 1316-1368 (формирование citations)
- `SourcesPanel.tsx`, строки 361-387 (отображение tabs)

**Причина**: 
```typescript
// В тексте Claude ставит [1]
const usedIndices = new Set<number>();
const citationMatches = fullContent.matchAll(/\[(\d+)\]/g);

// Но в "Источники" показываем ragContext.length
// А в "Цитаты" показываем citations.length (отфильтрованные)
```

**Корень**: Три независимых счетчика:
1. `[N]` в тексте ответа (что поставил Claude)
2. `ragContext.length` (все найденные чанки)  
3. `citations.length` (чанки, на которые ссылается ответ)

Они могут не совпадать — Claude может использовать [1], [3], [5] пропустив [2], [4].

---

### Проблема 3: Разрыв между "Источники" и "Цитаты"

**Местоположение**: `SourcesPanel.tsx`, строки 357-388

**Причина**: UI показывает ДВА бейджа, которые открывают ОДИН и тот же `SourcesPanel`:
```tsx
// Badge 1: "3 источника" → открывает SourcesPanel с tab="sources"
// Badge 2: "1 цитата" → открывает ТОТ ЖЕ SourcesPanel с tab="citations"
```

Но данные почти идентичны:
- `ragContext[]` = текстовый формат чанков
- `citations[]` = структурированный формат ТЕХ ЖЕ чанков

**Корень**: Нет семантического различия. "Источник" = документ из которого взят контекст. "Цитата" = тот же документ, но в другом формате.

---

### Проблема 4: Ссылка указывает не на тот документ

**Местоположение**: 
- Claude prompt (строки 729-745) — нет валидации
- `rerank-chunks/index.ts` — оценивает релевантность, но не точность

**Причина**: Claude получает 10 чанков и сам решает какой использовать. Если в контексте нет "Правила №482", Claude может:
1. Ссылаться на другой документ, упоминающий эти правила
2. Процитировать из общих знаний, но поставить [1] на ближайший источник

**Корень**: Нет верификации "утверждение X действительно содержится в источнике N".

---

### Проблема 5: Отсутствие источника для заявленного документа

**Местоположение**: `chat-stream/index.ts`, строки 328-337 (FTS search)

**Причина**: Если документ не загружен в базу знаний (folder_ids), FTS его не найдет. Claude использует общие знания и ставит сноску [1] на первый доступный источник.

**Корень**: Нет механизма проверки "если документ X упомянут в ответе, он должен быть в citations".

---

### Проблема 6: Ссылка ведёт на массив/подборку фрагментов

**Местоположение**: `chat-stream/index.ts`, строки 550-589

**Причина**: Для split-документов (например, PDF разбитый на части) все части группируются:
```typescript
const groupedChunks = new Map<string, RankedChunk[]>();
for (const chunk of rankedChunks) {
  const groupKey = chunk.parent_document_id || chunk.id;
  // Все части одного документа под одним номером
}
```

**Корень**: Один `[1]` может ссылаться на несколько чанков из одного документа, без указания какой именно фрагмент подтверждает тезис.

---

### Проблема 7: Открываемый фрагмент не релевантен утверждению

**Местоположение**: `SourcesPanel.tsx`, строки 151-224 (openDocumentWithHighlight)

**Причина**: При клике на источник передаётся `search_keywords` из ОРИГИНАЛЬНОГО запроса, а не из конкретного утверждения:
```typescript
searchText: citationData.search_keywords?.length 
  ? citationData.search_keywords.join(' ')
  : cleanSearchText(citationData.content_preview)
```

**Корень**: Все цитаты имеют ОДИНАКОВЫЕ `search_keywords` (из запроса пользователя), а не специфичные для каждого утверждения.

---

### Проблема 8: Отсутствие явной привязки "утверждение → конкретный фрагмент"

**Местоположение**: Архитектурная проблема — нигде не реализовано

**Причина**: Система работает так:
1. Claude получает чанки [1], [2], [3]...
2. Claude сам решает где ставить ссылки
3. Backend просто парсит [N] из текста и показывает соответствующий чанк

**Корень**: Нет структуры данных типа:
```typescript
interface InlineCitation {
  id: string;                    // Уникальный ID этой сноски
  text_span: [number, number];   // Позиция в тексте ответа
  assertion: string;             // Конкретное утверждение
  source_chunk_id: string;       // ID чанка, подтверждающего утверждение
  source_quote: string;          // Конкретная цитата из чанка
}
```

---

### Проблема 9: Подмена источников внутри одной сноски

**Местоположение**: `chat-stream/index.ts`, строки 1346-1363

**Причина**: `citations[]` формируется из `rankedChunks[]` с последовательной нумерацией:
```typescript
const allCitations = rankedChunks.map((chunk, idx) => ({
  index: idx + 1,  // [1], [2], [3]...
  document: chunk.document_name,
  // ...
}));
```

Но если rankedChunks содержит несколько чанков из разных документов под одним parent_document_id, они все получают разные номера, хотя логически это "один источник".

**Корень**: Смешение понятий "документ" и "чанк". Один документ = один источник, но много чанков.

---

### Проблема 10: Неопределённость идентификаторов источников

**Местоположение**: 
- `chat-stream/index.ts`, строки 571-584 (формирование ragContext)
- UI — нигде не показывает полную информацию

**Причина**: Формат источника:
```typescript
citation = `[${idx + 1}] ${displayName}`;
if (chunk.part_number) citation += ` (часть ${chunk.part_number}/${chunk.total_parts})`;
if (chunk.section_title) citation += ` | ${chunk.section_title}`;
if (chunk.article_number) citation += ` | Статья ${chunk.article_number}`;
```

Но в UI показывается только `citation.document` без section/article.

**Корень**: Метаданные есть, но не используются в кликабельных ссылках в тексте ответа.

## Предлагаемый план исправлений

### Фаза 1: Улучшение инструкций Claude (быстрое исправление)

**Файл**: `supabase/functions/chat-stream/index.ts`

Обновить инструкции:
```typescript
instructions = `
ПРАВИЛА ЦИТИРОВАНИЯ:
1. Каждое утверждение должно иметь ОДНУ конкретную ссылку [N]
2. Если утверждение основано на нескольких источниках — укажи все: [1][3]
3. НЕ повторяй одну ссылку [1] для разных фактов — это затрудняет проверку
4. Формат: "Утверждение [N]." — ссылка ПОСЛЕ утверждения
5. Если документ отсутствует в источниках — НЕ ссылайся на него`;
```

### Фаза 2: Разделение "Источники" и "Цитаты" по смыслу

**Файлы**: 
- `SourcesPanel.tsx` — изменить логику отображения
- `chat-stream/index.ts` — добавить разделение данных

Концепция:
- **Источники** = уникальные документы (deduplicated by document_id)
- **Цитаты** = конкретные фрагменты, использованные в ответе (matched by [N])

### Фаза 3: Кликабельные ссылки в тексте

**Файлы**: `ChatMessage.tsx`, `DepartmentChatMessage.tsx`

Рендерить `[N]` как кликабельные элементы:
```tsx
// В ReactMarkdown components:
text: ({ children }) => {
  // Парсим [1], [2] и заменяем на <button onClick={...}>[1]</button>
}
```

### Фаза 4: Специфичные search_keywords для каждой цитаты

**Файлы**: `chat-stream/index.ts`

Вместо одних search_keywords для всех цитат — извлекать контекст из самой цитаты:
```typescript
search_keywords: extractKeywordsFromChunk(chunk.content, message)
```

### Фаза 5: Верификация источников (долгосрочная)

**Архитектурное изменение**: 

После генерации ответа — второй вызов Claude для валидации:
```typescript
// Для каждого [N] в ответе проверить:
// 1. Утверждение действительно содержится в источнике N
// 2. Источник N существует в citations
// 3. Если нет — пометить как "не подтверждено"
```
