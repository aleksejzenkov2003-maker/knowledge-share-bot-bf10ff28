

# Исправление форматирования Excel-таблиц для RAG

## Проблема

Сейчас Excel-файл МКТУ (тысячи строк с 5 колонками) превращается в сплошной текст вида:

```
Строка 1: Класс/Class: 1; Базовый номер/ Basic Number: 10001; ... Строка 2: Класс/Class: 1; ...
```

Чанкер разрезает этот текст на куски по 3000 символов в **произвольных местах** -- посередине строки таблицы. В результате чанки получаются нечитаемыми, и RAG не может нормально найти нужную запись.

## Решение

Переформатировать вывод XLSX-парсера так, чтобы каждая запись таблицы была **отдельным блоком**, разделённым пустой строкой. Тогда чанкер будет разрезать между записями, а не внутри них.

### Было (текущий формат):
```
Строка 1: Класс/Class: 1; Базовый номер: 10001; Изменен ВОИС: ; RU_2026: окислители; EN_2026: combusting preparations Строка 2: ...
```

### Станет (новый формат):
```
---
Класс: 1 | No: 10001
RU: окислители [химические добавки для моторного топлива]
EN: combusting preparations [chemical additives to motor fuel]

---
Класс: 1 | No: 10002
RU: вещества клеящие для промышленных целей
EN: adhesives for industrial purposes
```

## Технические изменения

### Файл: `supabase/functions/process-document/index.ts`

**1. Переписать функцию `parseXLSXSheetToText`** (строки 1451-1521):
- Каждая строка таблицы форматируется как отдельный блок с разделителем `---`
- Пустые значения пропускаются
- Заголовки колонок сокращаются для экономии токенов (убираются дублирующие пояснения в скобках из заголовков)

**2. Улучшить чанкер для Excel** (строка 2446):
- Вместо `chunkTextSimple(text, 3000)` использовать специализированную функцию `chunkExcelText`, которая разбивает текст **только по разделителям записей** (`---`), а не по произвольным символам
- Каждый чанк будет содержать целые записи таблицы, без обрезки посередине

**3. Добавить новую функцию `chunkExcelText`:**
- Разбивает текст по разделителю `---`
- Группирует записи в чанки до 3000 символов
- Гарантирует, что ни одна запись не будет разрезана пополам
- Добавляет заголовок с названием файла и колонок в начало каждого чанка для контекста

### Ожидаемый результат

- Каждый чанк будет содержать 15-30 целых записей таблицы (вместо обрезанного текста)
- RAG сможет точно находить конкретные товары/услуги по запросу
- Поиск по номеру класса МКТУ или базовому номеру будет работать корректно

Если файл содержит больше 200000 строк его нужно дробить
