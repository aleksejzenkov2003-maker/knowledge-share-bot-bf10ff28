
## Что не так

Edge Function `deep-research` падает с **`CPU Time exceeded`** через ~37 секунд (видно в логах). Причина — модель `sonar-deep-research` присылает гигантский поток `<think>` (десятки тысяч токенов рассуждений), а наш SSE-парсер делает `JSON.parse` + `toLowerCase` + поиск тегов на каждой дельте. CPU квоту инстанса мы выжигаем раньше, чем модель успевает выдать финальный ответ. Поэтому фронт получает 0 байт и показывает «сервер прервал выполнение».

## Гибридное решение

1. **Снять нагрузку с CPU функции (минимизировать обработку каждой дельты)**
   - Не делать `JSON.parse` на каждом чанке. Использовать быстрый `indexOf('"content":"')` и доставать дельту строкой; полноценный `JSON.parse` — только для строк с `"citations"` или `finish_reason`.
   - Пропускать `<think>` блоки до того, как они попадут в `JSON.stringify` и обратно в SSE наружу: считать только символы вне `<think>`, ничего не пересылать пользователю до закрытия блока (чтобы не плодить 500+ мелких SSE-сообщений в секунду).
   - Эмитить дельты пользователю **батчами раз в 250 мс** (накапливать в `pendingDelta`), а не на каждый принятый кусочек.

2. **Авто-фоллбэк при сбое (главное по выбору пользователя — «Гибрид»)**
   - В блоке `catch` (а также если за 60 секунд не пришёл ни один контент-токен) — автоматически делаем второй вызов с моделью `sonar-reasoning-pro` (web-search + chain-of-thought, но в разы быстрее и без CPU-разрыва), стримим её результат.
   - В чате выводим короткий префикс: `_⚠️ Глубокое исследование недоступно, переключаюсь на быстрый анализ с веб-поиском..._` — чтобы пользователь видел, что произошло.
   - В метаданных проставляем `fallback_used: 'sonar-reasoning-pro'` для последующего анализа.

3. **Сократить вход для deep-research**
   - Передавать в `sonar-deep-research` **только последнее сообщение пользователя** (без всей истории чата) — это снижает входные токены и время до первого ответа в 2–3 раза. История остаётся для фоллбэка.
   - `max_tokens` для deep-research уменьшить с 4000 до **2500** (Perplexity всё равно режет ответ финальным саммари).

4. **Корректные сообщения об ошибках на фронте**
   - Если итоговый контент пустой и фоллбэк тоже не сработал — показывать конкретно: «Превышено время CPU у функции исследования» вместо общего «сервер прервал выполнение».
   - В `useOptimizedChat.ts` и `useChat.ts` подхватить новое поле `metadata.fallback_used` и показывать бейдж модели уже не `sonar-deep-research`, а той, что реально ответила.

## Файлы к изменению

- `supabase/functions/deep-research/index.ts` — лёгкий парсер SSE, батчинг 250 мс, авто-фоллбэк на `sonar-reasoning-pro`, передача только последнего user-message, `max_tokens=2500`, новое поле `fallback_used` в metadata.
- `src/hooks/useOptimizedChat.ts` — читать `metadata.fallback_used`, показывать корректную ошибку при пустом ответе.
- `src/hooks/useChat.ts` — то же.
- (опционально) `src/components/chat/ChatMessage.tsx` — мелкий бейдж `Fallback: sonar-reasoning-pro`, если поле есть.

## Технические детали

- Никаких миграций БД, никаких новых таблиц/очередей. Всё остаётся в одном edge-вызове, просто внутри него — две попытки.
- Heartbeat (10с) и hard-timeout (350с) сохраняются.
- Совместимо с текущим SSE-протоколом фронта (`type: 'content' | 'metadata'`).

## Итог для пользователя

После исправления:
- В большинстве случаев `sonar-deep-research` сам успеет ответить (мы даём ему больше CPU-бюджета);
- Если не успеет — мгновенно подхватит `sonar-reasoning-pro` и пользователь получит исследование с веб-источниками + явное предупреждение о фоллбэке;
- «Бесконечной загрузки» / «сервер прервал выполнение» больше не будет.
