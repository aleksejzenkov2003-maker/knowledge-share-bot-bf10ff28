

# Многоагентная система проектной работы — план реализации

## Что описано в ТЗ

Система превращает текущий «Проекты» из простого мультиагентного чата в **пошаговый workflow-движок**, где:
- Проект содержит **цепочку этапов** (steps), каждый привязан к агенту
- Каждый этап: агент выполняет работу → пользователь видит результат → может отредактировать → подтвердить → данные передаются следующему агенту
- В финале — сборка итогового документа из результатов всех этапов

Первый сценарий: **Подготовка КП по товарному знаку** (6 вкладок: Входные данные → МКТУ → Охраноспособность → Конфликты → Поиск в открытых источниках → Итог).

## Текущее состояние

Сейчас «Проекты» — это мультиагентный чат с @mention, контекст-пакетами и памятью. Нет пошагового workflow, нет этапов, нет конструктора цепочек.

## Что нужно построить

### Фаза 1: Инфраструктура workflow-движка

**1. Новые таблицы БД (миграция)**

- **`workflow_templates`** — шаблоны рабочих цепочек
  - `id`, `name`, `description`, `created_by`, `is_active`, `created_at`
  
- **`workflow_template_steps`** — шаги шаблона
  - `id`, `template_id` (FK), `step_order` (int), `name`, `description`, `agent_id` (FK → chat_roles), `input_schema` (jsonb — какие данные ожидает), `output_schema` (jsonb), `is_user_editable` (bool), `auto_run` (bool — запускать автоматически или ждать пользователя), `created_at`

- **`project_workflows`** — запущенный workflow в проекте
  - `id`, `project_id` (FK), `template_id` (FK), `status` (enum: draft/running/paused/completed), `created_by`, `created_at`, `completed_at`

- **`project_workflow_steps`** — конкретные шаги запущенного workflow
  - `id`, `workflow_id` (FK), `template_step_id` (FK), `step_order`, `status` (enum: pending/running/completed/error/skipped), `agent_id` (FK → chat_roles), `input_data` (jsonb — входные данные от предыдущего шага или пользователя), `output_data` (jsonb — результат работы агента), `user_edits` (jsonb — правки пользователя), `started_at`, `completed_at`, `error_message`

- **`project_step_messages`** — чат-лог внутри каждого шага (для итераций/уточнений)
  - `id`, `step_id` (FK), `user_id`, `message_role`, `content`, `metadata` (jsonb), `created_at`

- RLS: доступ только участникам проекта

**2. Edge Function `workflow-step-execute`**

- Принимает `step_id`, берёт агента шага, input_data (из предыдущего шага + правки пользователя)
- Формирует контекст: системный промпт агента + input_data + контекст-пакеты проекта + память проекта
- Вызывает `chat-stream` логику (SSE стриминг)
- Сохраняет результат в `output_data`
- Обновляет статус шага

**3. Типы TypeScript** (`src/types/workflow.ts`)

- `WorkflowTemplate`, `WorkflowTemplateStep`, `ProjectWorkflow`, `ProjectWorkflowStep`, `StepStatus`, `WorkflowStatus`

### Фаза 2: UI проектного workflow

**4. Переработка страницы проекта**

Текущая страница `ProjectChat` остаётся как один из режимов. Добавляется новый режим — **Workflow**:

- **Карточка проекта** — верхняя панель с названием, статусом, прогресс-баром workflow
- **Горизонтальные вкладки** — по одной на каждый шаг workflow (как в ТЗ: «Входные данные», «МКТУ», «Охраноспособность» и т.д.)
- **Содержимое вкладки шага**:
  - Статус шага (ожидание / выполняется / завершён)
  - Результат агента (markdown с возможностью редактирования)
  - Чат с агентом для уточнений внутри шага
  - Кнопки: «Запустить шаг» / «Перезапустить» / «Подтвердить и передать дальше»
  - Просмотр входных данных от предыдущего шага

**5. Новые компоненты**:

- `WorkflowStepper` — горизонтальный stepper с статусами шагов
- `WorkflowStepView` — содержимое одного шага (результат + редактор + чат + кнопки)
- `WorkflowStepChat` — мини-чат для итераций внутри шага
- `WorkflowResultEditor` — редактируемое поле результата шага (markdown editor)
- `WorkflowProgress` — прогресс-бар

**6. Хук `useProjectWorkflow.ts`**:

- Загрузка workflow и шагов проекта
- Запуск шага (вызов edge function)
- Сохранение правок пользователя
- Подтверждение шага → передача output в input следующего
- SSE стриминг результата шага

### Фаза 3: Первый шаблон — КП по товарному знаку

**7. Seed-данные: шаблон «КП по товарному знаку»**

6 шагов с привязкой к существующим или новым агентам:
1. Приём входных данных (пользователь заполняет форму / загружает файлы)
2. Анализ МКТУ (агент подбирает классы)
3. Анализ охраноспособности (агент проверяет обозначение)
4. Анализ конфликтности (агент ищет сходные знаки)
5. Поиск в открытых источниках (агент с web search)
6. Итоговая сборка (агент компонует КП)

**8. Создание специализированных агентов** (через chat_roles):

- Агент «МКТУ-Аналитик» — с промптом для подбора классов МКТУ
- Агент «Охраноспособность» — с промптом анализа обозначения
- Агент «Конфликтолог» — с промптом анализа сходных знаков
- Агент «Веб-исследователь» — с `allow_web_search: true`
- Агент «Сборщик КП» — с промптом финальной компоновки

### Фаза 4: Финальная сборка документа

**9. Edge function `workflow-compile-result`**:

- Собирает output_data всех шагов
- Формирует итоговый структурированный документ
- Генерирует PDF через агента

## Порядок реализации

Предлагаю начать с **Фазы 1 + 2** (инфраструктура + UI), без привязки к конкретному шаблону КП. Это даст универсальный движок, к которому потом можно подключать любые цепочки.

1. Миграция БД (таблицы + enum'ы + RLS)
2. Типы TypeScript
3. Edge function для выполнения шага
4. Хук `useProjectWorkflow`
5. UI компоненты workflow
6. Интеграция в страницу проекта
7. Шаблон КП + агенты (отдельный шаг)

## Технические детали

- Workflow engine полностью на стороне клиента + edge functions, без дополнительного backend
- Стриминг результатов шагов через тот же SSE-механизм `chat-stream`
- Шаблоны workflow хранятся в БД → можно добавлять новые через админку
- Результат каждого шага — JSON с markdown-контентом, который можно отрендерить и отредактировать
- `settings` поле в `projects` таблице будет хранить `workflow_id` привязанного workflow
